# Codex 超长线目标 Spec：Storyworks 全面完善工程

> 适用场景：用于 Codex 的超长线 goal 模式。  
> 核心要求：一轮修复后如果没有阻塞，继续检查、拓展、验证、修复，不因单次构建通过或局部功能可用而提前结束。  
> 当前项目基线日期：2026-06-27。  
> 当前主要技术栈：FastAPI + SQLite + Vue 3 + Vite + Vuetify 3。  

---

## 1. 可直接用于 `/goal` 的目标描述

请你以超长线模式持续完善 Storyworks 项目。当前项目是一个面向修仙/互动叙事创作的多模块统一应用，包含项目管理、世界书、人物卡、剧本、预设、AI 生成与迭代、跨模块引用、版本历史、导出和 demo 数据。你的任务不是只修一个显眼 bug，而是反复执行“深度检查 → 发现问题 → 设计修复 → 实现 → 验证 → 继续拓展检查”的循环，直到项目达到可长期维护、可真实创作、功能闭环严密、体验稳定的状态。

必须遵守：

- 不要在一轮修复或一次构建通过后停止。若没有明确阻塞，继续检查下一类问题。
- 每轮都要基于当前工作区证据，不凭记忆判断完成。
- 当前已有前端体验整体方向是工作台式、克制、实用、Vuetify 原生控件优先；禁止引入多余装饰卡片、大圆角、营销式页面或破坏布局密度。
- 每个功能必须考虑前端、后端、数据库、跨模块引用、版本、AI 过程反馈、错误处理和验证。
- 不允许为了容易通过测试而缩小目标。任何修改都必须让真实创作能力更完整。
- 旧模块可以保留作为兼容参考，但统一应用应成为主线。不要删除用户已有文件或回滚非本任务变更。
- 只有当逐项验收证据覆盖本 spec 的所有完成条件，才允许标记 goal 完成。

## 3. 总体问题诊断

### 3.1 架构问题

当前统一应用已经可运行，但后端和前端都处于“可用原型向产品化系统过渡”的阶段。

必须长期解决：

- 后端 `main.py` 过大，路由、业务逻辑、AI prompt、数据转换、导出逻辑混在一起。
- 前端 `App.vue` 过大，模块页面、对话框、状态、请求、图谱和导出逻辑混在一个组件里。
- 数据库 schema 由字符串集中定义，缺少迁移版本管理。
- 旧模块和统一模块并存，缺少明确兼容策略、迁移策略和主线说明。
- demo 初始化直接写入生产数据库，缺少 demo 重置、导入、导出和环境区分。

### 3.2 数据与业务逻辑问题

必须长期检查：

- 所有 JSON 字段需要一致的结构定义、默认值、校验和前后端类型。
- 跨模块引用需要强一致：角色引用世界书、剧本引用角色/世界书、世界书和角色显示被剧本引用。
- 删除操作需要明确级联、软删除或阻止删除规则。
- 版本历史需要能对比、恢复、显示差异，而不是只存快照。
- AI 迭代保存应记录 prompt、结果、目标板块、操作者、版本摘要。
- demo 数据需要可重复验证，不应污染用户真实项目。

### 3.3 AI 体验问题

当前已经加入动态过程提示和迭代接口，但还不是完整 AI 工作流。

必须长期完善：

- AI 请求需要真正可流式显示或至少阶段化显示、日志持久化、失败可恢复。
- 每次 AI 生成/迭代都应支持“预览 → 接受 → 局部应用 → 保存版本 → 回滚”。
- 角色生成必须始终完整覆盖 9 个通用核心分类：基础（BASIC）、信息（KNOWLEDGE）、隐秘（SECRETS）、属性（ATTRIBUTES）、关系（RELATIONS）、物品（INVENTORY）、技能（SKILLS）、命运（FORTUNE）、补充（EXTRAS）。
- 世界书生成必须尊重分类模板和结构化字段。
- 剧本生成必须参考上文、角色、世界书和预设。
- 预设生成必须能整理、组合、应用到其他模块。
- AI 失败时不能丢失用户输入。

### 3.4 前端体验问题

必须长期检查：

- 页面不能堆叠装饰卡片；继续使用工作台式三栏/双栏布局。
- 对话框过多时需要统一交互规范。
- 所有保存/删除/生成/迭代必须有明确 loading、成功、失败和可重试状态。
- 长文本编辑需要更好的 Markdown 编辑/预览能力；世界书正文、剧本正文和 Markdown 型 AI 生成/迭代结果必须在编辑时同步提供渲染预览。
- JSON 编辑是临时方案，长期应替换为结构化表单，并保留高级 JSON 模式。
- 图谱目前是基础 SVG，后续应增强为可筛选、可编辑、可定位、可缩放。
- 移动端或窄屏需要可用的响应式折叠布局。

### 3.5 测试与质量问题

必须长期解决：

- 后端缺少 pytest 测试体系。
- 前端缺少组件测试和端到端测试。
- 当前没有统一 lint/format/typecheck gate。
- 构建通过不等于业务可用，需要 API 流程测试和真实 UI 测试。
- 数据库迁移、demo 初始化、AI mock、导出内容都需要可重复验证。

## 4. 长线执行循环

每一轮 Codex 执行必须遵循以下循环。

### 4.1 轮次入口检查

每轮开始先执行：

- 查看 `git status --short`，识别用户已有变更，不得回滚。
- 检查统一前后端是否仍为主线。
- 快速读取最近修改的核心文件。
- 运行必要的轻量验证，例如后端导入、关键接口、前端构建或相关局部测试。
- 根据当前状态更新计划。

### 4.2 问题发现

每轮至少选择一个高价值方向深入：

- 架构拆分。
- 某模块功能闭环。
- AI 工作流。
- 数据一致性。
- 前端体验。
- 测试覆盖。
- demo 数据与迁移。
- 性能和可维护性。

发现问题时必须记录：

- 问题现象。
- 影响范围。
- 触发路径。
- 需要修改的前端、后端、数据库或测试。
- 验证方法。

### 4.3 修复与拓展

执行修复时遵守：

- 小步可验证。
- 不做无关重构。
- 优先消除真实用户流程断点。
- 后端业务和数据库约束优先于前端临时绕过。
- UI 不要新增装饰性复杂度。
- 每个新增 API 都要考虑错误响应、数据校验和前端消费方式。

### 4.4 验证

每轮至少执行与改动相关的验证：

- 后端：导入、compile、API 流程测试、数据库计数或 pytest。
- 前端：`npm run build`，必要时用 Playwright 或浏览器请求验证。
- 数据：检查 demo/project 数据不重复、不丢失、不破坏用户数据。
- AI：优先使用 mock 或非真实调用验证契约；真实调用只在必要时进行。

### 4.5 继续条件

只要未满足最终完成条件，且没有连续三轮相同阻塞，必须继续：

- 修复下一个模块。
- 拓展测试。
- 拆分架构。
- 补齐边界处理。
- 提升 UI 工作流。
- 强化数据一致性。

## 5. 全局产品约束

### 5.1 UI 约束

- 使用 Vue 3 + Vuetify 3。
- 使用 Vuetify 原生组件和 Material Design Icons。
- 保持工作台体验：导航、列表、详情、编辑、状态反馈清晰。
- 避免多余卡片、重边框、大圆角、营销式 hero、大面积装饰背景。
- 工具按钮优先使用图标，并必须具备原生 `title` 与必要的 `aria-label`；避免同时使用自定义浮层和浏览器原生提示造成双重标签。
- 每个可点击按钮都必须能在悬浮时显示清晰标签；当前长期约束是保留下方浏览器原生 `title` 提示，不再叠加 Vuetify/custom tooltip。
- 复杂表单使用 tabs、expansion panels、dense controls，而不是堆叠卡片。
- 所有按钮必须有 loading/disabled/失败反馈。
- 长文本不应遮挡、溢出或导致布局跳动。
- Markdown 预览区域必须有稳定高度和滚动能力，避免长文撑乱对话框；预览是功能性编辑面板，不应额外引入装饰卡片。
- 所有模块至少支持桌面 1366 宽度，后续支持窄屏。

### 5.2 后端约束

- FastAPI 路由应模块化：`routers/projects.py`、`routers/worldbook.py`、`routers/characters.py`、`routers/scripts.py`、`routers/presets.py`、`routers/ai.py`。
- 业务逻辑应进入 services。
- 数据转换应进入 schemas 或 serializers。
- 数据访问应集中封装，避免拼接 SQL 分散在路由中。
- 所有写操作必须校验 project ownership。
- 所有跨模块引用必须验证目标存在。
- 删除操作必须处理引用关系。
- 版本记录必须在重要实体变更时写入。
- AI 调用必须有超时、错误处理和日志。

### 5.3 数据库约束

- 当前可继续使用 SQLite，但必须引入 schema version。
- 每次 schema 变更要可迁移，不得要求用户删库。
- JSON 字段必须有默认值和结构约定。
- 常用查询字段必须有索引。
- 跨模块引用要有一致查询方式。
- demo 数据必须 idempotent，可重复初始化。
- 真实项目数据不得被 demo reset 误删。

### 5.4 AI 约束

- 所有 AI 生成必须返回过程日志。
- 所有 AI 迭代必须支持目标板块。
- 所有 AI 结果必须能预览、接受、保存、回滚。
- AI 预览结果被应用时不得强制二次调用模型；应保存用户确认的预览结果，并记录 `apply` 审计日志、目标板块/字段、prompt、用户要求、过程日志和版本摘要。
- AI 结果不可信，必须解析、校验、补默认值。
- 角色生成不得把 prompt 当 name。
- 生成内容不得默认截断；需要提高 token、拆步或分段续写。
- 若 AI 失败，保留用户输入和已生成片段。
- 支持 mock AI，以便测试不依赖真实服务。
- LLM API 配置必须在设置界面可读取和保存；API Key 不得在前端接口中明文回显，保存时留空应保留原密钥，清除必须显式操作。
- AI 生成接口必须读取保存后的同一份运行时配置，不能缓存旧的空 API Key；`baseUrl` 需要兼容传入 `/v1` 或完整 `/chat/completions` 路径。

### 5.5 测试约束

- 新增后端逻辑要有 pytest 或 TestClient 覆盖。
- 新增前端关键流程要有构建验证，逐步引入 Playwright。
- 每个模块至少有 CRUD、AI mock、导出、版本、引用相关测试。
- 测试必须可重复执行。
- 不得用“接口存在”代替“业务闭环验证”。

## 6. 模块目标：项目管理

### 6.1 目标

项目管理应成为所有模块的统一入口，支持真实多项目创作。

必须实现：

- 创建、编辑、删除、归档项目。
- 项目级设置：题材、默认风格、AI 参数、模块启用状态、默认可见性。
- 项目概览：世界书数量、角色数量、剧本节点数量、预设数量、最近更新、待处理 AI 草稿。
- 项目导入/导出。
- demo 项目重置，不影响用户项目。
- 项目级搜索入口。

### 6.2 约束

- 项目删除默认软删除或二次确认。
- 任何模块数据必须绑定 project_id。
- 项目切换必须清空或刷新当前模块选择状态。
- 不允许跨项目读取或修改实体。

### 6.3 验收

- 能创建两个项目，数据互不污染。
- 删除或归档项目不影响其他项目。
- demo reset 后数量恢复到指定基线。
- 项目概览准确显示计数。

## 7. 模块目标：世界书

### 7.1 目标

世界书应支持大型修仙世界观长期维护。

必须实现：

- 无限层级分类。
- 分类模板与自定义字段。
- 条目 CRUD。
- 条目结构化字段编辑。
- Markdown 正文编辑与预览。
- 标签、状态、重要度、可见性、创建者。
- 条目版本历史、版本对比、版本恢复。
- 世界书关系：来源、影响、相关、对立，自定义标签和强度。
- 图谱视图：筛选、定位、点击详情、创建/删除关联。
- AI 生成新条目。
- AI 迭代已有条目。
- 指定板块局部重生成，例如定义、规则、冲突、钩子。
- AI 自动分类推荐。
- 世界书被剧本引用显示。
- 世界书被角色关联显示。
- JSON/Markdown 导出。

### 7.2 约束

- 删除分类时不得误删条目，除非用户明确选择级联。
- 删除条目时必须处理关系和引用。
- 关系不能指向自身。
- AI 生成必须使用分类模板和同分类上下文。
- 局部迭代必须只替换目标板块，不破坏其他内容。
- 图谱不应因大量节点卡死，后续应分页或按需加载。

### 7.3 验收

- demo 世界书至少 10 条，内容详细。
- 能创建分类、编辑模板、创建条目、建立关系。
- 能对已有条目的指定板块 AI 迭代并保存版本。
- 图谱能显示节点和关系。
- 剧本引用世界书后，世界书详情能看到引用位置。

## 8. 模块目标：人物卡

### 8.1 目标

人物卡应成为角色设定、关系、秘密与玩家可见信息的核心系统。

必须实现：

- 角色 CRUD。
- 9 个固定通用分类，不再随题材扩散大类：
  - 基础（BASIC）/ `basic`。
  - 信息（KNOWLEDGE）/ `knowledge`。
  - 隐秘（SECRETS）/ `secrets`。
  - 属性（ATTRIBUTES）/ `attributes`。
  - 关系（RELATIONS）/ `relations`。
  - 物品（INVENTORY）/ `inventory`。
  - 技能（SKILLS）/ `skills`。
  - 命运（FORTUNE）/ `fortune`。
  - 补充（EXTRAS）/ `extras`。
- 小字段必须在这 9 个大类内重新设计为通用结构，并以中文标签 + 英文键展示，例如 `姓名（name）`、`背景经历（background）`。
- `extras` 只存纯文本补充信息，不使用结构化对象；其余 8 个大类可根据修仙、都市、科幻等背景自适应解释。
- 不再鼓励新增大类；特殊题材优先映射到 `attributes`、`skills`、`fortune`、`extras` 等通用字段。
- Developer Mode 完整展示。
- Player Mode 根据字段可见性展示。
- 字段可见性配置界面。
- 角色模板。
- 从已有角色复制结构。
- 角色关系 CRUD。
- 关系类型预设与自定义。
- 关系方向、数值、事件历史。
- 关系图谱。
- 世界书关联：手动选择和 AI 推荐。
- 角色被剧本引用展示。
- AI 完整生成角色。
- AI 分步生成和迭代。
- 指定分类或字段局部重生成，例如 background、motivation、secret、skills。
- JSON/Markdown 导出。

### 8.2 约束

- AI 生成角色不得将用户提示词当 name。
- 角色生成必须补齐 9 个通用核心分类；即使 AI 返回旧结构，也要规范化并补默认值。
- 修仙、都市、科幻等题材只改变字段内容，不改变字段契约；例如修仙的功法/境界、都市的职业能力/社会资源、科幻的权限/植入体都归入 `skills`、`attributes`、`fortune`。
- Player Mode 默认不泄露 secrets 和 fortune。
- 字段可见性不能丢失 developer_data。
- 删除角色必须处理关系和剧本引用。
- 关系图谱必须避免重复边和自环。

### 8.3 验收

- `修仙背景-demo` 角色至少 5 个，结构完整且互有关联；`都市背景-demo` 与 `科幻背景-demo` 至少各有可验证的核心角色、世界书、剧本、预设和引用关系。
- 每个 demo 角色都必须具备 9 个通用核心分类，Markdown 导出必须显示中文大类名和英文键。
- AI 生成角色后，接受并保存能创建真实角色。
- 能对角色指定字段 AI 迭代并保存。
- Player Mode 和 Developer Mode 内容不同且符合可见性配置。
- 导出 Markdown 包含主要分类。

## 9. 模块目标：剧本

### 9.1 目标

剧本模块应支持从总纲到场景片段的长篇创作。

必须实现：

- 层级结构：总纲、卷、篇章、章节、场景、片段。
- 节点 CRUD。
- 拖拽排序和移动层级。
- Markdown 编辑与预览。
- 摘要、状态、标签、重要度、故事时间、地点。
- 自动保存草稿。
- 版本历史、对比、恢复。
- 引用世界书和角色。
- 引用双向显示。
- AI 生成总纲、章节、场景、片段。
- AI 迭代已有节点。
- 指定板块局部重生成，例如对白、动作、环境、结尾钩子。
- 参考上文、引用资料、角色设定和预设。
- 导出 JSON、Markdown、纯文本、剧本格式。

### 9.2 约束

- 删除父节点必须明确是否级联删除子节点。
- 移动节点不能形成循环。
- AI 场景生成必须参考关联角色和世界书。
- 剧本引用不能指向不存在实体。
- 导出应按树顺序稳定输出。

### 9.3 验收

- demo 剧本至少 3 个节点，且有父子结构。
- 场景节点能引用角色和世界书。
- 角色详情能看到剧本出场。
- 世界书详情能看到剧本引用。
- AI 迭代场景并保存后，版本历史可恢复。

## 10. 模块目标：预设

### 10.1 目标

预设模块应成为 AI 生成风格和结构约束的统一来源。

必须实现：

- 预设 CRUD。
- 维度编辑。
- 自定义文本块。
- 应用场景配置。
- 编译完整提示词。
- AI 生成预设。
- AI 整理预设。
- AI 迭代预设。
- 预设组合。
- 预设应用到世界书、人物卡、剧本生成。
- JSON/Markdown 导入导出。
- 从文本提取预设。

### 10.2 约束

- compiled_prompt 必须由 dimensions、custom_blocks、application_scenes 稳定生成。
- 用户可手工覆盖 compiled_prompt，但应提示与结构化配置不一致。
- 预设应用必须明确 scene_type。
- 组合预设时覆盖规则必须透明。

### 10.3 验收

- demo 预设至少 10 个。
- 能创建预设、编辑维度、编译提示词。
- 能将预设应用到 AI 生成接口。
- 能组合两个预设并得到可读结果。
- AI 整理返回 issues 和 fixed_preset。

## 11. 模块目标：AI 工作台

### 11.1 目标

AI 不应只是按钮，而应是全项目可追踪、可迭代、可审计的创作助手。

必须实现：

- 统一 AI 对话框或工作台。
- 动态过程反馈。
- 生成日志。
- 结果预览。
- 接受、局部接受、丢弃。
- 生成历史。
- 质量检查。
- 根据建议重新生成。
- AI 请求取消或超时处理。
- AI mock 模式。
- 分段续写，避免长内容截断。

### 11.2 约束

- 真实 API Key 不应写入仓库。
- AI 调用错误必须展示可理解信息。
- 生成结果必须可保存为版本。
- AI 日志不能无限增长，需要归档或限制。
- AI prompt 应能回看，便于复现。

### 11.3 验收

- 任意模块 AI 生成时，用户能看到执行过程。
- 任意实体 AI 迭代时，可指定板块。
- AI 失败后用户输入仍保留。
- mock AI 测试可在无网络时通过。

## 17. Demo 数据目标

当前 demo 基线包含三套固定背景样例：`修仙背景-demo`、`都市背景-demo`、`科幻背景-demo`。三者不是装饰样例，而是长期用于验证不同题材能否共用同一套世界书、人物卡、剧本、预设、AI 和导出契约。

必须保持：

- `修仙背景-demo` 保持高密度主样例：至少 10 条世界书、5 个角色、3 个剧本节点、10 个预设，并保留复杂关系与引用。
- `都市背景-demo` 保持现实/都市调查样例：至少包含完整世界书分类、详细世界书条目、核心角色、剧本节点、预设和剧本引用。
- `科幻背景-demo` 保持科幻/技术伦理样例：至少包含完整世界书分类、详细世界书条目、核心角色、剧本节点、预设和剧本引用。
- 三套 demo 都必须使用同一套 9 分类人物卡契约，不得为某一题材新增专属大类。
- 世界书、人物、剧本、预设内容必须详细、互相一致，并能证明字段模型可适配修仙、都市、科幻三类差异很大的创作背景。
- demo seed 必须幂等，可重复运行，不重复插入，不破坏用户真实项目。

后续拓展：

- 增加一个更大 demo，用于性能测试。
- 增加 demo reset 按钮。
- 增加 demo export。
- 增加 demo consistency check。
- 增加三套 demo 的自动验收统计视图，明确展示每套 demo 的世界书、角色、剧本、预设、引用和版本数量。

## 18. 每轮优先级策略

当不知道下一步做什么时，按以下顺序选择：

1. 修复会阻断保存、生成、引用、删除、导出的 bug。
2. 补齐测试和验证。
3. 拆分单文件架构。
4. 完善 AI 工作流。
5. 完善模块功能闭环。
6. 提升 demo 和数据一致性。
7. 优化性能和 UX 细节。
8. 清理文档和启动体验。

## 19. 禁止事项

- 禁止因为构建通过就宣称整体完成。
- 禁止只做前端假数据绕过后端。
- 禁止删除用户旧模块或数据，除非用户明确要求。
- 禁止把 AI prompt 当实体名称。
- 禁止隐藏错误或吞掉异常。
- 禁止让 JSON 字段结构漂移。
- 禁止新增与现有设计风格冲突的卡片化、营销式页面。
- 禁止无验证地修改数据库 schema。
- 禁止真实 AI 测试成为唯一验证方式。

## 21. 建议的首轮执行任务

若从本 spec 开始新的超长线 goal，建议第一轮执行：

1. 建立后端 pytest 基础和 TestClient fixtures。
2. 为 demo seed 数量、项目隔离、角色保存、世界书迭代、剧本引用、预设组合写测试。

5. 若无问题，继续拆人物卡模块。

这条路线优先把“可验证”和“可维护”立住，后续大修不容易把已有功能弄坏。
