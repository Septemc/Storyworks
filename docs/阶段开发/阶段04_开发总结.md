## 2. 本阶段目标

完成剧本模块的最小可用闭环，包括剧本基础数据结构、结构化编辑、大纲与场景卡管理、剧本草稿生成、导出与基础校验能力。

## 3. 实际完成内容

- 后端新增剧本 Schema、Repository、Service、Validator、API 路由。
- 实现剧本 CRUD 接口。
- 实现剧本草稿生成接口。
- 实现剧本 JSON / Markdown 导出接口。
- 实现剧本结构基础校验，特别是场景卡关键字段校验。
- 前端新增剧本页面 `ScriptsPage.vue`。
- 前端新增剧本 Schema 与数据服务。
- 前端实现项目切换、剧本列表、结构化编辑、草稿生成、保存、导出和删除交互。
- 前端导航接入剧本模块入口。
- 首页当前阶段目标展示更新。
- 新增剧本后端测试，并通过测试验证。

## 4. 未完成内容

- 当前章节与场景卡采用 JSON 文本编辑，尚未实现可视化结构树或专用卡片编辑器。
- 当前剧本生成仍是规则化草稿生成，尚未接入真实大模型。
- 尚未实现剧本与角色卡、世界书条目的显式引用联动。
- 尚未实现分支图可视化和章节拖拽排序。

## 5. 关键实现说明

- 剧本结构统一收束为：总纲字段 + 主要节点 + 分支设想 + 章节结构 + 场景卡 + 约束对象。
- 后端沿用前几阶段已经形成的 `Model / Schema / Repository / Service / Validator / Route` 分层方式，保持实现风格一致。
- 场景卡使用结构化 JSON 列表表示，最小强制字段包括：`title / goal / location / participants / must_happen`。
- 剧本草稿生成优先保证结构可用和可继续编辑，不追求复杂剧情推演。
- 导出统一支持 `json` 与 `markdown`，便于作为后续创作资料继续使用。

## 6. 问题与解决方式

- 问题：剧本结构既需要有总纲字段，也需要能表达章节和场景卡，若拆得过细会让第四阶段过重。
  - 解决方式：先采用“单文档 + 多个结构化块”的设计，把复杂可视化编辑延后到后续阶段。
- 问题：章节、场景、约束这类字段天然是复杂对象，前端若全部做专用 UI，会显著拖慢阶段进度。
  - 解决方式：当前先采用 JSON 文本编辑，保证结构稳定和闭环可用。
- 问题：剧本校验过弱会让数据结构失控，校验过强又会阻碍最小可用。
  - 解决方式：先只对标题、状态和场景卡关键字段做基础硬校验。

## 7. 当前遗留风险

- JSON 文本编辑对非技术型用户不够友好，后续需要更直观的编辑器。
- 剧本生成当前偏结构模板化，更适合作为草稿起点。
- 剧本与人物卡、世界书之间暂时没有正式引用约束，后续整合阶段需要补齐。
