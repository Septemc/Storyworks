# Storyworks 阶段开发任务书

版本：v1.0  
日期：2026-04-13  
适用范围：Storyworks 独立辅助生成器

## 1. 文档目标

本文档用于明确 Storyworks 的阶段开发路线、每阶段的工作范围、交付物、验收标准，以及阶段结束后的文档沉淀要求。

本项目所有阶段开发完成后，都必须在 `docs/阶段开发` 目录下补写对应的开发总结文档。

## 2. 阶段开发总原则

- 每一阶段都必须有明确范围，不做无限扩张。
- 每一阶段都必须有可演示的成果。
- 每一阶段都必须同步更新相关设计文档。
- 每一阶段都必须补写一份开发总结。
- 开发总结必须放在 `docs/阶段开发` 目录下。

## 3. 阶段文档要求

### 3.1 必须产出的文档

每个阶段结束后，至少要补齐以下文档：

- 阶段开发总结
- 阶段遗留问题清单
- 下一阶段衔接说明

### 3.2 开发总结命名规范

建议统一命名为：

- `阶段01_开发总结.md`
- `阶段02_开发总结.md`
- `阶段03_开发总结.md`
- `阶段04_开发总结.md`
- `阶段05_开发总结.md`
- `阶段06_开发总结.md`

### 3.3 开发总结必须包含的内容

- 本阶段目标
- 实际完成内容
- 未完成内容
- 关键实现方案
- 遇到的问题
- 解决方式
- 当前遗留风险
- 对下一阶段的建议

## 4. 阶段划分总览

本项目建议拆分为六个阶段：

1. 基础工程搭建阶段
2. 世界书模块阶段
3. 人物卡模块阶段
4. 剧本模块阶段
5. 预设模块阶段
6. 收尾整合与发布阶段

## 5. 阶段一：基础工程搭建阶段

### 5.1 阶段目标

完成项目最小可运行基础，包括前端应用、后端应用、基础项目结构和通用能力。

### 5.2 开发范围

- 建立 `apps/frontend`
- 建立 `apps/backend`
- 完成基础路由
- 完成基础页面框架
- 完成基础后端接口框架
- 完成基础配置加载
- 完成基础数据库初始化
- 完成统一错误处理
- 完成基础文档目录整理

### 5.3 交付物

- 可启动的前端项目
- 可启动的后端项目
- 项目首页骨架
- 基础健康检查接口
- 基础配置文件
- 项目架构文档初版

### 5.4 验收标准

- 前端可本地启动
- 后端可本地启动
- 前后端可基础联调
- `apps` 目录结构符合架构文档
- 开发文档目录可正常使用

### 5.5 阶段结束必须补写

- `docs/阶段开发/阶段01_开发总结.md`

## 6. 阶段二：世界书模块阶段

### 6.1 阶段目标

基于第一阶段已经完成的前后端基础工程，正式落地 Storyworks 的世界书模块，使项目具备“世界书条目创建、编辑、结构化生成、质量校验、导出复用”的完整闭环。

本阶段不追求复杂检索、知识召回、运行时上下文装配等高级能力，只聚焦世界书作为独立创作资产的生产与管理。

### 6.1.1 阶段完成后的用户结果

阶段二结束后，用户应能够：

- 在项目内进入世界书模块
- 查看世界书分类与条目列表
- 手动创建世界书条目
- 使用 AI 生成世界书草稿
- 编辑并保存世界书条目
- 对世界书条目进行基础结构校验
- 将世界书条目导出为 Markdown 或 JSON

### 6.1.2 阶段边界

本阶段明确纳入：

- 世界书分类与条目模型
- 世界书前后端 CRUD
- 世界书页面与编辑表单
- 世界书 AI 草稿生成
- 世界书导出
- 世界书基础校验

本阶段明确不纳入：

- 向量检索
- 条目自动关联推荐
- 复杂版本树
- 批量导入大型知识库
- 世界地图可视化编辑器
- 多人协同冲突处理

### 6.2 开发范围

#### 6.2.1 数据结构与领域定义

- 定义世界书分类枚举
- 定义世界书条目基础字段
- 定义世界书条目内容结构
- 定义世界书创建、更新、导出请求结构
- 定义世界书校验规则

#### 6.2.2 后端开发范围

- 新增世界书数据模型
- 新增世界书 Repository
- 新增世界书 Service
- 新增世界书 Validator
- 新增世界书 API 路由
- 新增世界书生成接口
- 新增世界书导出接口
- 新增世界书模块基础测试

#### 6.2.3 前端开发范围

- 新增世界书列表页
- 新增世界书详情/编辑页
- 新增世界书创建入口
- 新增世界书分类筛选
- 新增世界书表单组件
- 新增世界书生成弹窗或生成面板
- 新增世界书导出入口
- 新增世界书前端数据请求服务

#### 6.2.4 AI 辅助能力范围

- 根据一句世界观概述生成世界书蓝图草稿
- 根据分类和标题生成世界书条目草稿
- 根据已有简述扩写为标准条目
- 对已有条目进行规范化整理

#### 6.2.5 文档范围

- 更新项目全面设计文档中与世界书相关的落地说明
- 补充阶段二开发总结
- 如有必要补充世界书字段说明文档

### 6.3 交付物

### 6.4 验收标准

#### 6.4.1 功能验收

- 用户可在前端进入世界书模块
- 用户可创建世界书条目
- 用户可编辑并保存条目
- 用户可删除条目
- 用户可按分类查看条目
- 用户可生成世界书草稿
- 用户可导出世界书条目

#### 6.4.2 结构验收

- 世界书条目结构与设计文档一致
- 必填字段缺失时能够提示
- 条目分类值受到限制
- 导出内容结构稳定

#### 6.4.3 工程验收

- 前后端可完成世界书模块基础联调
- 世界书接口具备基础测试或冒烟验证
- 世界书页面可本地运行并完成基础演示

### 6.5 世界书条目标准字段

本阶段建议世界书条目至少包含以下字段：

- `id`
- `project_id`
- `category`
- `title`
- `summary`
- `keywords`
- `content`
- `notes`
- `status`
- `created_at`
- `updated_at`

其中 `content` 建议采用结构化对象，至少支持：

- `definition`
- `origin`
- `structure`
- `rules`
- `impact`

说明：

- 并非每个条目都必须完整使用上述全部子字段，但至少要保证整体结构统一。
- 后续阶段若需要扩展世界书能力，应在本结构上向后兼容扩展。

### 6.6 世界书分类基线

本阶段建议内置以下默认分类：

- 历史
- 地理
- 政治
- 人文
- 风俗
- 科学
- 经济
- 宗教
- 阵营
- 制度

如开发过程中需要扩展，可补充：

- 种族
- 组织
- 事件
- 术语
- 技术体系
- 能力体系

### 6.7 页面与交互任务拆分

#### 6.7.1 世界书列表页

页面应至少支持：

- 当前项目世界书列表展示
- 按分类筛选
- 按标题搜索
- 新建条目入口
- 点击进入详情编辑

#### 6.7.2 世界书编辑页

页面应至少支持：

- 基础字段编辑
- 内容结构编辑
- 关键词编辑
- 草稿保存
- 手动校验
- 导出操作

#### 6.7.3 世界书生成交互

页面应至少支持：

- 输入一句概念或简要说明
- 选择分类
- 选择生成模式
  - 新建草稿
  - 扩写已有内容
  - 规范化已有内容
- 将生成结果写回编辑区

### 6.8 后端接口建议

本阶段建议至少提供以下接口：

- `GET /api/projects/{project_id}/worldbook`
- `POST /api/projects/{project_id}/worldbook`
- `GET /api/worldbook/{entry_id}`
- `PATCH /api/worldbook/{entry_id}`
- `DELETE /api/worldbook/{entry_id}`
- `POST /api/worldbook/generate`
- `GET /api/worldbook/{entry_id}/export`

说明：

- 如当前后端路由风格已有约定，可按已有规范调整命名。
- 导出接口至少支持 `markdown` 与 `json` 两种格式。

### 6.9 目录落位建议

结合第一阶段现有工程结构，本阶段建议新增或补充以下目录内容：

后端：

- `apps/backend/app/models/worldbook.py`
- `apps/backend/app/schemas/worldbook.py`
- `apps/backend/app/repositories/worldbook_repository.py`
- `apps/backend/app/services/worldbook_service.py`
- `apps/backend/app/validators/worldbook_validator.py`
- `apps/backend/app/api/worldbook.py`
- `apps/backend/app/prompts/worldbook/`
- `apps/backend/tests/test_worldbook.py`

前端：

- `apps/frontend/src/pages/WorldbookPage.vue`
- `apps/frontend/src/services/worldbook.ts`
- `apps/frontend/src/schemas/worldbook.ts`
- `apps/frontend/src/features/worldbook/`

### 6.10 阶段执行顺序建议

建议按以下顺序推进：

1. 先定义世界书 Schema 与分类常量
2. 完成后端数据模型与 CRUD
3. 完成前端列表与编辑页
4. 打通前后端联调
5. 接入 AI 草稿生成
6. 接入导出能力
7. 完成基础测试与文档补写

### 6.11 阶段风险提示

- 若一开始就追求复杂世界书结构，可能导致页面和接口实现过重。
- 若不先固定分类与字段，后续人物卡和剧本阶段会反复返工。
- 若生成结果没有统一结构，后续导出和复用会很混乱。

因此本阶段应坚持：

- 先统一结构
- 再做页面
- 再做生成
- 最后补齐导出与测试

### 6.12 阶段结束必须补写

- `docs/阶段开发/阶段02_开发总结.md`

## 14. 任务书结论

Storyworks 的开发应坚持“小步快跑、阶段验收、阶段总结、持续沉淀”的方式推进。

因此，本任务书的执行要求非常明确：

- 前后端统一放在 `apps` 目录下开发
- 按六个阶段推进
- 每个阶段完成后，必须在 `docs/阶段开发` 下新增一份开发总结
- 没有总结，不进入下一阶段
