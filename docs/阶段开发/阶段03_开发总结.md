## 2. 本阶段目标

完成人物卡模块的最小可用闭环，包括默认通用角色模板接入、模板驱动编辑、`developer_mode` / `player_mode` 双视图、人物卡生成、导出与基础防剧透校验。

## 3. 实际完成内容

- 后端接入默认通用角色模板，并提供模板读取接口。
- 后端新增人物卡数据模型 `CharacterProfile`。
- 后端新增人物卡 Schema、Repository、Service、Validator、API 路由。
- 实现人物卡 CRUD 接口。
- 实现人物卡草稿生成接口。
- 实现人物卡模板列表与模板详情接口。
- 实现人物卡 JSON / Markdown 导出接口。
- 实现基础 `player_mode` 防剧透校验，限制秘密与命运相关字段在玩家视角下必须保持“未知”。
- 前端新增人物卡页面 `CharactersPage.vue`。
- 前端新增人物卡模板 Schema 与人物卡数据服务。
- 前端实现项目切换、人物卡列表、模板驱动字段编辑、双视图切换、草稿生成、保存、导出和删除交互。
- 前端导航接入人物卡模块入口。
- 首页当前阶段目标展示更新。
- 新增人物卡后端测试，并通过测试验证。

## 4. 未完成内容

- 当前只接入了默认通用角色模板，尚未支持多模板管理。
- 当前模板驱动编辑器优先保证可用，尚未对复杂字段做更细粒度的专用控件。
- 当前人物卡生成仍采用规则化草稿生成，尚未接入真实大模型。
- 尚未支持人物卡批量生成、模板市场和高级关系图编辑。

## 5. 关键实现说明

- 模板读取直接复用 `docs/当前开发现状文档/角色模板/默认通用角色模板/角色模板-默认通用角色模板.json`，确保第三阶段不是重新定义人物卡标准，而是实际接入现有模板资产。
- 后端人物卡模块沿用第一、二阶段已建立的 `Model / Schema / Repository / Service / Validator / Route` 分层方式，保持代码组织一致。
- 人物卡编辑器采用模板驱动渲染：以前端读取模板中的 `tabs` 与 `fields` 后动态构建编辑界面，而不是把字段硬编码进页面。
- 双视图使用 `developer_mode` 与 `player_mode` 分离存储，并在保存前执行基础防剧透校验。
- 对复杂字段类型（如 `json`、`object_list`、`relation_graph`）当前采用文本编辑 + JSON 解析方式，优先保证模板全字段可编辑。

## 6. 问题与解决方式

- 问题：默认角色模板文件带有 UTF-8 BOM，后端按普通 `utf-8` 读取时会报错。
  - 解决方式：模板读取改为 `utf-8-sig`。
- 问题：模板字段类型较多，若一次性为每种字段做专用编辑器，第三阶段会明显膨胀。
  - 解决方式：先统一采用模板驱动表单，复杂字段使用 JSON 文本编辑作为最小可用实现。
- 问题：`player_mode` 对复杂字段也需要支持“未知”占位，否则保存阶段 JSON 解析会失败。
  - 解决方式：前端在提交前对包含“未知”的值做字符串保留，不强制解析为 JSON 对象。

## 7. 当前遗留风险

- 复杂字段使用 JSON 文本编辑，用户体验仍有优化空间。
- 防剧透校验目前是基础规则，尚未覆盖更细粒度的“已知信息逐步揭示”场景。
- 当前人物卡生成偏模板化，更适合作为草稿起点，不适合直接作为高质量终稿。
