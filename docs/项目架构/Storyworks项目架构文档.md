# Storyworks 项目架构文档

版本：v1.0  
日期：2026-04-13  
适用范围：Storyworks 独立辅助生成器

## 1. 项目定位

Storyworks 是一个完全独立的创作辅助生成器，聚焦四类核心资产：

- 世界书
- 人物卡
- 剧本
- 预设

它的职责是帮助用户创建、编辑、生成、校验、整理与导出这四类资产，而不是承担完整的叙事运行时、检索引擎或主项目集成平台职责。

## 2. 架构目标

本项目架构需要满足以下目标：

- 前端和后端都放在 `apps` 目录下。
- 支持单机开发与独立部署。
- 支持结构化资产编辑，而不是只依赖自然语言输入。
- 支持 AI 辅助生成，但所有结果都可人工修订。
- 支持模板化、版本化、导出和复用。
- 支持后续逐步扩展，但第一阶段保持架构简单清晰。

## 3. 总体目录规划

建议采用以下顶层结构：

```text
Storyworks/
├─ apps/
│  ├─ frontend/
│  └─ backend/
├─ docs/
│  ├─ 项目架构/
│  ├─ 阶段开发/
│  └─ 当前开发现状文档/
├─ reference_code/
├─ scripts/
├─ test/
└─ temp/
```

其中，核心要求如下：

- 所有前端应用代码放在 `apps/frontend`
- 所有后端应用代码放在 `apps/backend`
- 项目文档放在 `docs`
- 临时导出、预览文件和调试产物放在 `temp`

## 4. `apps` 目录设计

### 4.1 前端目录

建议前端目录结构如下：

```text
apps/frontend/
├─ public/
├─ src/
│  ├─ app/
│  ├─ router/
│  ├─ pages/
│  ├─ features/
│  ├─ components/
│  ├─ stores/
│  ├─ services/
│  ├─ schemas/
│  ├─ utils/
│  ├─ styles/
│  └─ main.ts
├─ index.html
├─ package.json
└─ vite.config.ts
```

各目录职责建议如下：

- `app`
  - 全局应用初始化
  - 布局
  - 全局 providers
- `router`
  - 页面路由定义
  - 权限入口
  - 路由守卫
- `pages`
  - 页面级模块
  - 按业务页面拆分
- `features`
  - 业务功能模块
  - 可复用业务逻辑与局部 UI
- `components`
  - 通用组件
  - 表单控件
  - 弹窗
  - 面板
- `stores`
  - 全局状态
  - 会话状态
  - 当前项目状态
- `services`
  - 请求封装
  - API 调用
  - 文件上传下载
- `schemas`
  - 前端表单结构
  - 字段映射
  - 校验规则
- `utils`
  - 通用工具函数
- `styles`
  - 全局样式
  - 主题变量

### 4.2 后端目录

建议后端目录结构如下：

```text
apps/backend/
├─ app/
│  ├─ api/
│  ├─ core/
│  ├─ domain/
│  ├─ services/
│  ├─ repositories/
│  ├─ schemas/
│  ├─ validators/
│  ├─ prompts/
│  ├─ adapters/
│  ├─ models/
│  ├─ tasks/
│  └─ main.py
├─ migrations/
├─ tests/
├─ requirements.txt
└─ pyproject.toml
```

各目录职责建议如下：

- `api`
  - HTTP 路由
  - 请求响应模型转换
  - 控制器层
- `core`
  - 配置
  - 日志
  - 错误处理
  - 常量
  - 应用启动逻辑
- `domain`
  - 世界书领域
  - 人物卡领域
  - 剧本领域
  - 预设领域
  - 模板领域
  - 导出领域
- `services`
  - 具体业务编排
  - 生成流程组织
  - 草稿转正式逻辑
- `repositories`
  - 数据访问
  - 文件读取写入
  - 持久化抽象
- `schemas`
  - Pydantic 请求响应结构
  - 资产结构 Schema
- `validators`
  - 模板校验
  - 结构校验
  - 业务规则校验
- `prompts`
  - AI 生成提示模板
  - 资产生成规则
  - 风格约束文本
- `adapters`
  - LLM provider 适配
  - 文件存储适配
  - 格式转换适配
- `models`
  - ORM 模型
  - 数据表模型
- `tasks`
  - 异步任务
  - 批量生成
  - 导出任务

## 5. 前端架构设计

### 5.1 技术建议

前端建议使用：

- Vue 3
- TypeScript
- Vite
- Pinia
- Vue Router
- Element Plus 或 Naive UI

选择 Vue 的原因：

- 表单驱动页面开发效率高
- 更适合结构化编辑器和配置型后台
- 对多标签页、多面板、多表单场景更友好

### 5.2 前端页面结构

建议前端页面至少包括：

- 项目首页
- 世界书管理页
- 人物卡管理页
- 剧本管理页
- 预设管理页
- 模板管理页
- 导入导出页
- 设置页

### 5.3 页面模块拆分

#### 项目首页

职责：

- 展示当前项目概览
- 展示四类资产统计
- 展示最近编辑记录
- 展示最近生成记录

#### 世界书管理页

职责：

- 世界书分类浏览
- 世界条目编辑
- AI 生成世界条目草稿
- 世界条目规范化
- 世界条目导出

#### 人物卡管理页

职责：

- 角色模板选择
- 人物卡新建与编辑
- `developer_mode` / `player_mode` 双视图切换
- AI 生成角色草稿
- 角色卡校验与导出

#### 剧本管理页

职责：

- 剧本大纲编辑
- 章节与场景结构编辑
- 剧情约束编写
- AI 补全场景草稿
- 剧本导出

#### 预设管理页

职责：

- 预设列表浏览
- 预设创建与编辑
- 文风与参数配置
- 预设测试
- 预设导出

#### 模板管理页

职责：

- 角色模板管理
- 世界书模板管理
- 剧本模板管理
- 字段配置与版本管理

### 5.4 前端 `features` 模块建议

建议按业务域拆分：

```text
apps/frontend/src/features/
├─ project/
├─ worldbook/
├─ character/
├─ script/
├─ preset/
├─ template/
├─ generation/
└─ export/
```

每个 feature 内建议包含：

- 页面局部组件
- 表单模型
- 请求函数
- 类型定义
- 校验器

### 5.5 前端状态管理建议

建议使用以下 Store：

- `useProjectStore`
  - 当前项目
  - 当前选中资产
- `useTemplateStore`
  - 模板列表
  - 模板缓存
- `useGenerationStore`
  - 最近生成记录
  - 生成状态
  - 生成错误
- `useSettingsStore`
  - 全局配置
  - LLM 参数
  - 界面设置

### 5.6 前端组件体系建议

建议沉淀以下通用组件：

- 结构化表单渲染器
- JSON 编辑器
- 双视图切换组件
- 标签页编辑器
- 版本差异对比组件
- 资产预览卡片
- 生成结果审阅面板
- 导出配置弹窗

## 6. 后端架构设计

### 6.1 技术建议

后端建议使用：

- Python 3.11+
- FastAPI
- Pydantic
- SQLAlchemy
- SQLite 作为第一阶段默认数据库

后续如果规模扩大，可再扩展 PostgreSQL，但第一阶段不必引入复杂数据库架构。

### 6.2 后端分层原则

后端建议采用简单清晰的四层：

1. API 层
2. Service 层
3. Repository 层
4. Storage 层

其目标是避免一开始做过重的服务化拆分。

### 6.3 API 层职责

API 层负责：

- 接收前端请求
- 参数校验
- 调用对应服务
- 统一返回结果
- 统一异常处理

建议路由划分如下：

```text
apps/backend/app/api/routes/
├─ health.py
├─ projects.py
├─ templates.py
├─ worldbooks.py
├─ characters.py
├─ scripts.py
├─ presets.py
├─ generations.py
├─ exports.py
└─ settings.py
```

### 6.4 Service 层职责

Service 层负责业务编排，不直接处理底层存储细节。

建议服务划分如下：

- `ProjectService`
- `TemplateService`
- `WorldbookService`
- `CharacterService`
- `ScriptService`
- `PresetService`
- `GenerationService`
- `ExportService`

### 6.5 Repository 层职责

Repository 层负责：

- 资产读写
- 版本存储
- 查询过滤
- 文件元数据记录

建议按资产类型拆分：

- `ProjectRepository`
- `TemplateRepository`
- `WorldbookRepository`
- `CharacterRepository`
- `ScriptRepository`
- `PresetRepository`
- `GenerationRepository`

### 6.6 Adapter 层职责

Adapter 层主要处理外部依赖适配。

建议包括：

- `llm`
  - OpenAI
  - 兼容 OpenAI 的本地模型接口
  - 其他未来模型适配
- `storage`
  - 本地文件存储
  - 导出包写入

### 6.7 Validator 层职责

Validator 层是本项目的重点之一。

建议至少实现以下校验器：

- 世界书结构校验器
- 人物卡模板校验器
- `player_mode` 防剧透校验器
- 剧本场景结构校验器
- 预设参数校验器
- 导出前完整性校验器

## 8. 数据存储方案

### 8.1 第一阶段存储原则

第一阶段建议使用轻量方案：

- 业务主数据使用 SQLite
- 大段结构化内容使用 JSON 字段
- 导出文件使用本地文件系统

这更符合独立辅助生成器的定位。

### 8.2 推荐核心数据对象

建议最少包含以下对象：

- 项目
- 模板
- 世界书条目
- 人物卡
- 剧本
- 预设
- 生成记录
- 导出记录

### 8.3 统一资产元数据

四类核心资产建议共享以下元数据：

- `id`
- `project_id`
- `title`
- `summary`
- `status`
- `version`
- `tags`

### 8.4 文件导出策略

## 9. AI 生成流程设计

### 9.1 生成原则

Storyworks 的 AI 生成只服务于“资产草稿生成”，不承担完整剧情运行职责。

### 9.2 通用生成流程

建议统一采用以下流程：

1. 用户选择资产类型
2. 用户填写必要输入
3. 后端读取对应模板
4. 后端组合提示词与预设
5. 调用 LLM 生成草稿
6. 执行结构化校验
7. 返回前端审阅
8. 用户确认保存

### 9.3 四类生成入口

建议提供以下生成入口：

- 世界书条目生成
- 人物卡生成
- 剧本大纲生成
- 预设草稿生成

### 9.4 生成功能边界

生成器只做以下事情：

- 生成结构化草稿
- 补全文本描述
- 规范化字段内容
- 修复格式错误

生成器不做以下事情：

- 代替用户完成整套创作决策
- 运行完整剧情模拟
- 自动维护复杂长期记忆系统
- 构建检索增强系统

## 11. 安全与配置设计

### 11.1 配置范围

建议后端统一管理以下配置：

- LLM provider
- API key
- 默认模型
- 默认温度参数
- 导出目录
- 数据库路径

### 11.2 安全原则

- 密钥只保存在后端配置中
- 前端不直接暴露模型密钥
- 导出目录必须可配置
- 文件上传类型需要白名单限制

## 12. 开发规范建议

### 12.1 前端规范

- 页面和功能模块分离
- 组件命名统一
- 表单 Schema 与页面逻辑解耦
- 请求层统一封装

### 12.2 后端规范

- 路由层不写复杂业务
- Service 层负责编排
- Repository 层不泄露业务规则
- Prompt 与业务逻辑解耦
- 校验器独立可测试

## 13. 第一阶段推荐落地方式

为避免架构过重，建议按以下方式起步：

- 先搭建 `apps/frontend` 和 `apps/backend`
- 先实现项目、模板、世界书、人物卡四条主线
- 剧本和预设在第二阶段补齐完整能力
- 第一阶段默认本地 SQLite
- 第一阶段默认本地文件导出

## 14. 架构结论

Storyworks 的项目架构不应一开始设计成大型平台，而应围绕“独立辅助生成器”目标构建。

因此，最合适的架构是：

- 前端放在 `apps/frontend`
- 后端放在 `apps/backend`
- 采用单体应用、清晰分层、轻量存储、模板驱动、结构化生成的方式逐步推进

这样既能满足当前功能目标，也能为后续演进留出空间。
