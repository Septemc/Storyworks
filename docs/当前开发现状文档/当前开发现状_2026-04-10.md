# Storyteller 当前开发现状（按 2026-04-11 代码实况重写）

## 1. 文档说明

本文档基于 2026-04-11 仓库中的实际代码、目录结构、运行入口与测试集重写。

- 文件名沿用 `当前开发现状_2026-04-10.md`，但内容已不再以 2026-04-10 的旧结论为准。
- `docs/当前开发现状文档/当前开发现状_2026-04-10.md` 现在应被视为“当前主线现状文档入口”，而不是历史快照。
- 项目内部分 README 仍保留早期 Phase-2 ~ Phase-7 的阶段表述，这些表述可以作为演进线索参考，但不能替代当前代码事实。

## 2. 当前项目定位

Storyteller 当前已经不是单一的“Phase-7 双阶段故事控制台样板”，而是一个：

- 以 `apps/` 七层架构为主干；
- 以 `gateway_frontend` 为统一 FastAPI/ASGI 入口；
- 以 PostgreSQL 为默认运行底座；
- 以 BusinessBook + messaging 为跨层调用约束；
- 同时承载桌面前端、移动前端、管理后台、兼容 API、故事生成链路、世界书检索链路、预设路由、模型选择与后台管理能力

的一体化叙事系统主线仓库。

当前代码仍普遍使用 “Phase-7” 作为架构代号，但从能力范围看，主线已经进入：

**Phase-7 主线完成后的扩展整合阶段**，重点不再是“把链路搭起来”，而是“让全量前端、兼容接口、运行时策略、后台能力与数据库实现真正落在同一主线上”。

## 3. 目录与代码组织现状

### 3.1 顶层结构

当前顶层主目录包括：

- `apps/`：主运行层代码
- `runtime/`：网关调用的运行时编排与工具模块
- `packages/`：前端 SDK 与共享契约包
- `scripts/`：诊断脚本与 CI/smoke 脚本
- `test/`：正式测试集
- `docs/`：架构、重构、预设与现状文档
- `data/`、`assets/`、`temp/`：运行时或资源目录

### 3.2 `apps/` 七层主干仍然成立

当前 `apps/` 下保留并实际运行的分层为：

- `gateway_frontend`
- `business_core`
- `narrative_engine`
- `memory_knowledge`
- `worldbook_factory`
- `shared_protocol`
- `messaging`

边界规则仍然是项目共识：

- 跨层请求必须走 BusinessBook
- 跨层通信必须走 messaging
- `business_core` 是唯一数据库读写执行层
- `user_id` / `session_id` 作用域校验是硬约束

对应守卫脚本仍在使用：

- `scripts/ci/check_import_rules.py`
- `scripts/ci/check_file_length.py`
- `scripts/ci/check_database_connectivity.py`

### 3.3 各层当前职责

`gateway_frontend`

- 项目统一 ASGI 入口：`apps.gateway_frontend.asgi:app`
- 负责健康检查、能力快照、静态前端托管、认证、管理台、兼容 API、Phase-7 trace 接口、世界书与故事接口聚合
- 当前已演变为项目最大的集成层，`apps/gateway_frontend/asgi.py` 超过 12000 行，是全仓库最重的单文件之一

`business_core`

- 仍是唯一数据读写执行层
- 当前不只是早期内存仓储，已经同时存在：
  - `SqlBusinessRepository`
  - `SqlFrontendDataRepository`
  - `InMemoryBusinessRepository`
- SQL 仓储已经承载故事、角色、模板、预设、正则、LLM 配置、世界书、消息公告等大量真实读写逻辑

`narrative_engine`

- 已具备完整两阶段链路：
  - Stage-1：正文生成
  - Stage-2：变量推理与状态写回
- 包含外部模型调用适配、变量命令解析、Stage-1/Stage-2 服务实现
- 已接入输出格式约束、重试与 provider 适配逻辑

`memory_knowledge`

- 已具备世界书检索闭环
- 已具备独立记忆任务闭环：
  - `memory.index.rebuild`
  - `memory.vectorize`
  - `dynamic_worldbook_create`
  - `dynamic_worldbook_revise`

`worldbook_factory`

- 仍为独立能力层
- 当前已存在 `blueprint / entries / generate / normalize / revise` 等运行时接口支撑
- 网关侧保留 `experimental/worldbook-factory/*` 入口

`shared_protocol`

- 不只是 BusinessBook/Envelope/路由枚举
- 还承载了当前运行时推理状态相关基础协议与算法模块，包括：
  - `session_analysis_state.py`
  - `runtime_analysis_scheduler.py`
  - `runtime_preference_governor.py`
  - `runtime_relationship_governor.py`
  - `runtime_relationship_character_tracks.py`
  - `story_segment_format.py`
  - `worldbook_light_retrieval.py`
  - `official_preset_assets.py`

`messaging`

- 仍提供主题目录、序列化、producer/consumer、重试、死信与 trace 传播能力
- 目前仍以进程内 transport 形态服务主线集成

## 4. 运行入口与运行模式现状

### 4.1 统一启动入口

当前推荐入口仍是：

```bash
python -m apps --host 127.0.0.1 --port 8020 --reload
```

等价入口：

```bash
python -m uvicorn apps.gateway_frontend.asgi:app --host 127.0.0.1 --port 8020 --reload
```

启动脚本仍位于仓库根目录：

- `start_backend.ps1`
- `start_backend.bat`
- `start_backend.sh`

### 4.2 数据库模式已是默认主路径

当前网关启动时会在 `startup` 阶段执行数据库检查：

- 若设置了 `STORYTELLER_SKIP_DB_CHECK=1`，则跳过数据库检查并进入测试/兜底路径
- 默认情况下如果未配置 `NOVEL_DATABASE_URL`，应用会直接启动失败
- 成功启动后，`/healthz` 与 `/gateway/capabilities` 会返回数据库连接状态、数据库名、当前用户与版本信息

结论：

- **真实 PostgreSQL 模式是当前主路径**
- **内存路径只是测试与兜底模式，不应再被描述为主实现**

### 4.3 网关是 no-proxy 模式

当前能力快照明确返回：

- `proxy_mode=disabled`
- `frontend_migrated=true`

这说明当前网关已经不是“给旧后端做代理”的前置层，而是：

- 自己托管前端静态资源
- 自己提供前端所需数据 API
- 自己承接大量旧接口兼容逻辑

## 5. 前端与托管形态现状

`packages/app_sdk` 已作为共享前端 SDK 存在，当前模块包括：

这说明当前项目已经进入“多前端共用同一网关与共享 SDK”的形态，而不是单一控制台原型。

## 6. 核心业务能力现状

### 6.1 故事生成主链路

当前故事主链路已经具备以下闭环：

1. 前端发起故事生成
2. 网关进入 `two_stage_runtime`
3. `narrative_engine` 执行 Stage-1 正文生成
4. 通过 `business_core` 写入分段结果
5. 写入后同步触发 Stage-2 变量推理
6. Stage-2 将结果写回变量快照、角色状态、玩家记忆、事件账本
7. 网关提供 trace 查询、详情查询与 Stage-2 retry

当前可用接口仍包括：

- `POST /api/phase7/story/start`
- `GET /api/phase7/traces`
- `GET /api/phase7/traces/{trace_id}`
- `POST /api/phase7/stage2/retry`

同时，面向旧前端的数据接口也仍保留：

- `GET /api/story/recent`
- `GET /api/story/saves/list`
- `GET /api/story/saves/detail`
- `POST /api/story/saves/create`
- `POST /api/story/generation_jobs/start`
- `POST /api/story/generation_jobs/latest/regenerate`
- `GET /api/story/generation_jobs/current`

结论：

- Phase-7 接口是“新主链路观测入口”
- `/api/story/*` 兼容接口仍是当前业务面上的重要组成部分，并未真正退场

### 6.2 预设、模式路由与运行时分析

当前项目已经形成比旧文档更完整的“运行时策略层”，主要体现在：

- `apps/gateway_frontend/preset_mode_router.py`
- `apps/gateway_frontend/session_runtime_preview.py`
- `apps/shared_protocol/session_analysis_state.py`
- `apps/shared_protocol/runtime_analysis_scheduler.py`

这部分能力已经覆盖：

- `story_mode` 归一化与标签路由
- 模型能力识别与模式选择
- 自动模式下的偏好治理与策略切换
- 关系状态、角色跟踪、叙事推进状态的增量分析
- 会话运行时偏好预览与持久化

这说明当前项目不再只是“给模型一段提示词然后返回文本”，而是在向“会话级叙事状态机 + 预设路由系统”演进。

### 6.3 LLM 运行时与故障恢复

当前运行时模块中已经明确存在：

- `runtime/llm_api_profiles.py`
- `runtime/llm_model_discovery.py`
- `runtime/output_format_constraint.txt`

结合测试可以确认，当前已经实际覆盖：

- provider base url 归一化
- 不同 provider 的 stream 兼容差异处理
- MiniMax 兼容与 provider hint 识别
- 模型列表拉取
- 余额不足/429 等场景下的 failover
- 对严格输出长度与结构契约的重试修复

这部分已经是主线正式能力，不应再被写成“近期修复中的临时适配”。

### 6.4 世界书与记忆链路

当前世界书相关能力分为三层：

1. `memory_knowledge`
   - 检索
   - 向量化
   - 独立任务闭环
2. `worldbook_factory`
   - 生成
   - 修订
   - 规范化
3. `gateway_frontend` 兼容 API
   - 世界书列表/详情/导入/删除
   - 语义搜索
   - rebuild chunks

同时，测试已经覆盖：

- 世界书轻量检索
- 远程 embedding / rerank
- legacy ready/reference id 兼容兜底
- 世界书写入合同
- worldbook factory runtime contract

### 6.5 管理后台能力已真正落地

当前 `web_admin` 与 `/api/admin/*` 已经落地到真实代码，而不是占位页面。

当前管理后台能力至少包括：

- 管理员登录与身份校验
- 系统概览
- 用户列表与用户详情
- 注册初始化策略（bootstrap settings）

- 站内消息 CRUD / publish / reset-reads
- 世界书管理、用户列表与导出

这意味着当前主线已经具备“系统后台”而不只是“玩家控制台”。

## 9. 与旧文档相比，最重要的认知更新

以下几点是本次重写中最需要纠正的旧认知：

1. `business_core` 不再只是“未来会换成持久层”的阶段性实现，SQL 仓储已经是主实现。
2. `gateway_frontend` 不再只是 Phase-7 控制台入口，而是整个项目的统一集成网关。
3. 前端不是只剩“一个控制台页面”，而是已经并入桌面端、移动端、管理端、落地页和 APK 工程。
4. 当前项目已形成预设路由、运行时分析、关系跟踪、模型能力适配等更高层运行时逻辑。
5. 管理后台、公告消息、LLM 公共配置、检索服务配置都已落到代码，而不是规划项。
6. `/api/story/*`、`/api/worldbook/*` 等兼容接口仍然是现役主路径的一部分，不能简单当成“即将废弃的历史残留”。

## 10. 当前风险与技术债

基于当前代码现状，最核心的风险仍然集中在以下几项：

1. **网关单文件过重**
- `apps/gateway_frontend/asgi.py` 已超过 12000 行
- 路由、认证、后台、兼容层、静态托管、数据库接入、运行时胶水逻辑高度耦合

2. **文档与实现存在阶段错位**
- 多个 README 仍停留在早期 Phase 描述
- 新人如果只看 README，容易低估当前系统复杂度，或误判哪些模块还是“计划中”

3. **兼容接口面过宽**
- 当前新链路与旧接口并存
- 对外行为虽然更稳定，但维护复杂度高，回归面大

4. **数据库路径与内存兜底长期并存**
- 测试友好，但会持续带来行为漂移风险
- 文档与测试必须持续明确哪条才是生产主路径

5. **进程内 messaging 形态仍限制未来多实例部署**
- 当前足以支撑本地与单实例主线
- 但如果后续进入多实例部署，需重新评估消息一致性、重试语义与状态同步方案

## 11. 下一阶段建议

### 11.1 首要建议

优先从 `gateway_frontend` 拆分责任边界，建议至少按以下方向拆：

- 认证与用户接口
- 管理台接口
- 故事/Phase-7 接口
- 世界书与资源接口
- 兼容 `/api/story/*` 接口
- 静态前端托管

### 11.2 第二优先级建议

统一刷新项目说明文档，至少同步以下文件：

- `README.md`
- `apps/README.md`
- `apps/business_core/README.md`
- `apps/narrative_engine/README.md`
- `apps/memory_knowledge/README.md`
- `apps/gateway_frontend/README.md`
- `test/README.md`

目标是让文档对“SQL 已成主路径、后台已落地、多前端已并入、运行时分析已存在”形成一致描述。

### 11.3 第三优先级建议

围绕高风险主链路继续收敛专项回归组：

- 故事生成与 regenerate
- recent / saves / latest segment 删除链路
- 跨用户权限拦截
- 公共模型配置与 failover
- 世界书检索 / rebuild / semantic search
- 公告消息投放与用户侧消费

## 12. 一句话结论

截至 2026-04-11，Storyteller 主线已经从“Phase-7 链路搭建完成”推进到“全量前端并入、真实数据库落地、管理后台上线、兼容接口共存、运行时策略增强”的一体化集成阶段；当前最主要的问题不再是功能缺失，而是网关复杂度、文档滞后与兼容面收敛。
