# Storyworks 项目全面设计文档

版本：v2.0  
日期：2026-04-13  
状态：重构版设计基线稿

> 说明：本文将 Storyworks 明确定义为一个完全独立的“世界书、人物卡、剧本、预设”辅助生成器，而不是大型主项目中的网关、运行时或集成中台。本文不讨论检索系统、主项目架构、复杂服务拆分或剧情运行时编排，只聚焦资产生成与创作辅助本身。

## 1. 项目定义

Storyworks 是一款面向互动叙事创作的独立辅助工具，用于帮助创作者生成、编辑、整理、约束和导出四类核心叙事资产：

- 世界书
- 人物卡
- 剧本
- 预设

它不是故事运行引擎，也不是在线游戏服务端，更不是复杂的多模块主站系统。它的价值在于把原本分散、松散、容易失控的叙事设定沉淀成结构清晰、格式统一、可重复生成、可人工修订的创作资产。

## 2. 产品定位

### 2.1 核心定位

Storyworks 的核心定位是：

- 为创作者提供结构化资产编辑能力
- 为创作者提供 AI 辅助生成能力
- 为创作者提供统一模板与规范能力
- 为创作者提供可导出、可复用的资产成果

### 2.2 一句话定义

Storyworks 是一个为互动叙事项目服务的结构化资产生成工坊。

### 2.3 非定位

Storyworks 不负责以下内容：

- 不负责在线剧情运行
- 不负责复杂会话记忆系统
- 不负责检索引擎设计
- 不负责主项目网关与服务治理
- 不负责游戏内实时变量调度
- 不负责大型团队协作平台化治理

## 3. 产品目标

### 3.1 总体目标

通过统一的结构模板与生成规范，帮助创作者稳定地产出可用的世界书、人物卡、剧本和预设资产，降低从“灵感”到“可用资料”的整理成本。

### 3.2 具体目标

- 让世界设定能被系统化拆分和组织
- 让人物卡能具备统一模板与明确生成标准
- 让剧本能够作为剧情约束文档存在
- 让预设能够控制文风、质量和偏好
- 让四类资产之间具备明确关联关系
- 让输出结果足够规范，能够继续被人类或其他系统使用

### 3.3 成功标准

- 创作者能在单个项目中完成四类资产的建立与维护
- AI 生成结果大部分情况下可以直接作为草稿使用
- 输出格式稳定，字段结构清晰
- 资产之间关系清楚，不会互相脱节
- 项目可导出为完整创作资料包

## 4. 设计原则

### 4.1 资产优先

Storyworks 的核心产物不是对话，而是资产。每次生成都应尽可能落成一份可保存、可再编辑、可导出的资产。

### 4.2 结构优先

四类资产都必须优先采用结构化表达。自然语言用于描述，结构字段用于约束。

### 4.3 模板优先

所有核心生成都应以模板为基础，不允许完全无约束的自由生成取代模板化设计。

### 4.4 可编辑优先

任何 AI 生成内容都必须允许人工修改，且应尽量保持字段清晰、段落明确、易于继续编辑。

### 4.5 一致性优先

同一项目中的世界设定、人物卡、剧本、预设必须尽量保持相互一致，避免出现设定冲突、人物失真或风格漂移。

### 4.6 辅助而非替代

Storyworks 的定位是辅助创作，不是替代创作者做一切决定。系统应负责整理、生成、规范和提示，而关键判断仍由创作者掌握。

## 5. 目标用户

- 互动小说作者
- 跑团主持人
- 世界观设定作者
- 文字游戏开发者
- 视觉小说编剧
- 需要建立角色卡与设定资料的独立创作者

## 6. 核心能力范围

Storyworks 只聚焦以下六类能力：

- 项目创建与管理
- 世界书生成与编辑
- 人物卡生成与编辑
- 剧本生成与编辑
- 预设生成与编辑
- 资产导出与复用

不纳入本期范围的能力：

- 在线故事续写引擎
- 对话式剧情实时推进系统
- 高级向量检索平台
- 多人实时协同编辑
- 复杂权限审批流

## 7. 产品结构

### 7.1 基本组织方式

一个 Storyworks 项目由以下内容组成：

- 项目基础信息
- 世界书集合
- 人物卡集合
- 剧本集合
- 预设集合
- 模板集合
- 导出包

### 7.2 推荐工作台结构

- 项目概览
- 世界书
- 人物卡
- 剧本
- 预设
- 模板
- 导出中心
- 设置

### 7.3 项目级概念

一个项目应表达一套完整创作方向，通常包括：

- 项目名称
- 题材类型
- 风格方向
- 核心世界观简介
- 主要叙事目标
- 默认预设
- 默认模板

## 8. 资产模型总览

### 8.1 四类核心资产

| 资产类型 | 用途 |
| --- | --- |
| 世界书 | 记录世界观、设定、规则和知识背景 |
| 人物卡 | 记录角色信息、关系、状态和命运线索 |
| 剧本 | 约束剧情结构、场景推进和信息揭示 |
| 预设 | 控制生成的文风、质量、偏好和表达方式 |

### 8.2 通用元数据

所有资产建议共享以下基础字段：

```json
{
  "id": "asset_xxx",
  "type": "character",
  "title": "林青溪",
  "summary": "表面温和实则极度谨慎的巡夜人",
  "status": "draft",
  "tags": ["主角团", "北境", "巡夜人"],
  "version": 1,
  "created_at": "2026-04-13T10:00:00Z",
  "updated_at": "2026-04-13T10:00:00Z"
}
```

### 8.3 统一状态

- `draft`：草稿
- `ready`：可用
- `final`：定稿
- `archived`：归档

## 9. 世界书设计

### 9.1 模块定位

世界书用于承载项目中的长期设定知识。它不是百科式堆料工具，而是为创作直接服务的设定资产库。

### 9.2 核心目标

- 建立统一世界观基础
- 固定关键设定事实
- 为人物卡生成提供背景
- 为剧本编写提供边界
- 为预设应用提供题材上下文

### 9.3 世界书内容范围

世界设定应至少支持以下分类：

- 历史
- 地理
- 政治
- 人文
- 风俗
- 科学
- 经济
- 宗教
- 阵营
- 制度

可扩展分类：

- 种族
- 语言
- 技术体系
- 能力体系
- 法律
- 组织
- 事件
- 地标
- 术语

### 9.4 世界书组织方式

建议采用“分类 + 条目”的方式组织。

每个世界书条目建议包含：

- 标题
- 分类
- 摘要
- 正文
- 关键词
- 关联条目
- 适用范围
- 备注

### 9.5 世界书条目标准结构

```json
{
  "entry_id": "world_xxx",
  "category": "政治",
  "title": "七城议会",
  "summary": "北境联邦的最高协商机构",
  "keywords": ["联邦", "议会", "北境"],
  "content": {
    "definition": "由七座主城共同组成的协商议会。",
    "origin": "战争结束后各城缔约成立。",
    "structure": "每城一席，重大议案需五席通过。",
    "impact": "决定联邦军事、财政和对外关系。"
  },
  "relations": ["world_xxy"],
  "notes": "适合作为政治线关键背景。"
}
```

### 9.6 世界书生成能力

Storyworks 在世界书模块中应支持：

- 从一句世界观概述生成世界蓝图
- 从分类生成条目草稿
- 从简略设定扩写为完整条目
- 把零散灵感整理成标准世界书格式
- 对已有条目做语言润色与结构规范化

### 9.7 世界蓝图

在正式生成条目前，系统应支持先生成世界蓝图。蓝图建议包含：

- 世界主题
- 时代背景
- 主要地区
- 核心冲突
- 主要阵营
- 主体制度
- 技术或力量体系
- 创作禁区与风格边界

### 9.8 世界书生成规范

- 世界书应优先写“确定事实”，少写含糊表达
- 条目必须能够单独阅读，不依赖过多上下文
- 同一项目中术语要统一
- 世界设定必须能服务角色与剧情，而不是只堆装饰信息
- 科学、宗教、制度、经济等条目应尽量写出“如何影响人物和故事”

### 9.9 世界书编辑模式

建议支持三种编辑方式：

- 手动创建
- AI 草拟
- 半结构化补全

### 9.10 世界书质量要求

- 分类正确
- 表述清晰
- 逻辑自洽
- 术语统一
- 对创作有实际使用价值

## 10. 人物卡设计

### 10.1 模块定位

人物卡是角色资料资产，不是单一简介页。它应同时承担角色描写、角色约束、角色生成和角色后续扩写的基础资料功能。

### 10.2 设计依据

人物卡设计应以现有“默认通用角色模板”为核心参考，并将其确立为默认标准模板之一。

### 10.3 人物卡目标

- 统一角色资料结构
- 支持详细人物设计
- 支持双视角信息分层
- 支持剧情约束引用
- 支持继续扩写和批量生成

### 10.4 默认人物卡标签页

建议继续采用当前默认模板的八个标签页：

- 基础信息
- 见闻设定
- 秘密过往
- 数值属性
- 人际关系
- 装备物品
- 技能功法
- 命运转机

### 10.5 默认人物卡字段范围

建议默认覆盖以下字段：

- 姓名
- 身份
- 修为等级或角色层级
- 简述
- 性格特征
- 外貌细节
- 背景故事
- 心理创伤
- 隐藏身份
- 核心数值
- 健康状态
- 关系网络
- 当前装备
- 随身物品
- 掌握技能
- 天命标签
- 因果提示

### 10.6 双视角模式

#### 10.6.1 developer_mode

`developer_mode` 用于记录完整角色真相，要求：

- 必须尽可能完整
- 不允许用大量“未知”逃避设计
- 可以写入秘密信息
- 应服务创作者和后续扩写

#### 10.6.2 player_mode

`player_mode` 用于记录玩家或主角当前已知的信息，要求：

- 只能出现已经明确获知的内容
- 未揭示内容必须标记为未知
- 禁止提前剧透
- 应作为对外展示版本

### 10.7 人物卡生成规范

- 必须严格遵循模板结构输出
- 所有字段都应有内容或合理占位
- 角色背景必须解释角色当前状态
- 角色性格必须能对应其行为倾向
- 技能、装备、层级必须符合世界设定
- 关系网络必须具备明确对象和关系性质
- 天命标签和因果提示必须具有剧情驱动价值
- developer_mode 与 player_mode 必须同时生成

### 10.8 人物卡生成输入

系统应支持以下生成输入方式：

- 一句话角色概念
- 多段角色描述
- 基于世界书生成角色
- 基于阵营生成角色
- 基于剧本场景生成配角
- 基于现有角色模板批量生成

### 10.9 人物卡输出要求

人物卡输出应具备以下特征：

- 可直接阅读
- 可直接继续编辑
- 可直接导出
- 可直接用于后续创作

### 10.10 人物卡示例结构

```json
{
  "character_id": "char_xxx",
  "template_id": "comprehensive_character_v1",
  "developer_mode": {
    "tab_basic": {},
    "tab_knowledge": {},
    "tab_secrets": {},
    "tab_attributes": {},
    "tab_relations": {},
    "tab_inventory": {},
    "tab_skills": {},
    "tab_fortune": {}
  },
  "player_mode": {
    "tab_basic": {},
    "tab_knowledge": {},
    "tab_secrets": "未知",
    "tab_attributes": {},
    "tab_relations": {},
    "tab_inventory": {},
    "tab_skills": {},
    "tab_fortune": "未知"
  }
}
```

### 10.11 人物卡质量要求

- 形象鲜明
- 设定完整
- 表达具体
- 不空泛
- 可用于剧情推进

## 11. 剧本设计

### 11.1 模块定位

剧本模块用于约束剧情发展，不负责在线自动推进剧情。它的作用是帮助创作者把剧情设计整理成可供写作、生成和复用的结构文档。

### 11.2 剧本目标

- 明确故事主线与支线
- 约束剧情节奏和推进方向
- 规定信息揭示顺序
- 固定关键事件和关键节点
- 为 AI 生成提供剧情边界

### 11.3 剧本层级

建议采用以下层级：

- 故事总纲
- 卷或篇章
- 故事弧
- 章节
- 场景

### 11.4 剧本基础结构

剧本文档建议至少包含：

- 标题
- 类型
- 核心主题
- 故事摘要
- 主要人物
- 关键冲突
- 故事阶段
- 主要节点
- 分支设想
- 禁止越界内容

### 11.5 场景卡设计

场景卡是剧本中最重要的结构单元之一。每张场景卡建议包含：

- 场景标题
- 场景目的
- 参与人物
- 场景地点
- 进入条件
- 场景冲突
- 必须发生的事件
- 可发生的事件
- 禁止发生的事件
- 结束条件
- 可揭示信息

### 11.6 剧本约束类型

剧本应支持以下几类约束：

- 硬约束
- 软约束
- 顺序约束
- 揭示约束
- 角色行为约束

### 11.7 剧本生成能力

Storyworks 在剧本模块中应支持：

- 从一句故事概念生成总纲
- 从总纲拆解篇章
- 从篇章拆解章节
- 从章节拆解场景卡
- 将零散剧情灵感整理为标准剧本文档
- 将已有剧情说明规范为统一格式

### 11.8 剧本生成规范

- 剧本必须清晰表达主线目标
- 剧本必须说明关键冲突
- 场景应具备明确作用
- 信息揭示必须有顺序意识
- 禁止事件必须写清楚
- 支线不能严重破坏主线

### 11.9 剧本与人物卡关系

剧本中的主要人物应尽量对应已有角色卡。若未建立人物卡，系统应给出提示，建议补齐角色资料。

### 11.10 剧本与世界书关系

剧本中的世界背景应尽量来源于世界书。若剧本使用了新的设定概念，系统应建议补建相应世界书条目。

### 11.11 剧本质量要求

- 结构清晰
- 冲突明确
- 节奏合理
- 可读可用
- 能作为写作约束文档使用

## 12. 预设设计

### 12.1 模块定位

预设模块用于控制生成风格与生成偏好。它不提供故事事实，只提供表达方向与质量倾向。

### 12.2 预设目标

- 控制文风
- 控制表达密度
- 控制剧情偏好
- 控制人物塑造偏好
- 控制输出质量要求

### 12.3 预设分类

建议支持以下分类：

- 文风预设
- 题材预设
- 剧情偏好预设
- 人物塑造预设
- 质量预设
- 输出格式预设

### 12.4 文风预设示例

- 古风正剧
- 轻小说风
- 黑暗悬疑
- 史诗奇幻
- 硬派科幻
- 日常治愈

### 12.5 预设核心字段

一个预设建议至少包括：

- 名称
- 适用范围
- 风格描述
- 用词倾向
- 句式倾向
- 描写密度
- 对白比例
- 节奏倾向
- 情感浓度
- 剧情偏好
- 人物刻画偏好
- 禁止表达
- 输出要求

### 12.6 预设生成能力

系统应支持：

- 从一句风格说明生成预设
- 从参考文本总结预设
- 从题材类型生成通用预设
- 对已有预设进行收束、增强或改写

### 12.7 预设生成规范

- 预设必须清楚描述风格
- 预设必须具备可执行性
- 预设不能写成模糊空话
- 预设应尽量能影响实际输出
- 预设之间应允许比较和替换

### 12.8 预设质量要求

- 风格明确
- 表达具体
- 指向清晰
- 不空泛
- 可稳定复用

## 13. 四类资产之间的关系

### 13.1 世界书与人物卡

世界书提供人物的时代、制度、阵营、地理和文化背景。人物卡则把这些背景落到具体角色身上。

### 13.2 世界书与剧本

世界书决定剧情能否成立，剧本决定剧情如何展开。

### 13.3 人物卡与剧本

人物卡决定角色是谁，剧本决定角色在什么节点出场、行动和冲突。

### 13.4 预设与其他三类资产

预设不改变世界事实，不替代人物设定，不取代剧本结构。预设的作用是控制“如何表达”以及“在生成时更偏向什么”。

### 13.5 项目关系原则

- 世界书定义背景边界
- 人物卡定义角色边界
- 剧本定义剧情边界
- 预设定义表达边界

## 14. 生成工作流

### 14.1 世界书工作流

1. 输入世界观概念
2. 生成世界蓝图
3. 选择分类补全条目
4. 人工修订
5. 定稿

### 14.2 人物卡工作流

1. 选择人物模板
2. 输入角色概念
3. 选择关联世界背景
4. 生成 developer_mode
5. 生成 player_mode
6. 人工修订
7. 定稿

### 14.3 剧本工作流

1. 输入故事概念
2. 生成故事总纲
3. 拆分篇章与章节
4. 补充场景卡
5. 绑定人物与设定
6. 人工修订
7. 定稿

### 14.4 预设工作流

1. 输入风格目标
2. 生成预设草稿
3. 调整文风与偏好字段
4. 保存为项目预设
5. 复用到世界书、人物卡或剧本生成中

## 15. 生成模式设计

### 15.1 新建模式

从零开始生成完整资产。

### 15.2 扩写模式

把一句话或简要提纲扩写为完整结构。

### 15.3 规范化模式

把零散文本整理成标准模板。

### 15.4 改写模式

在不改变核心事实的前提下重写表达。

### 15.5 补全模式

只补齐缺失字段，不重写完整内容。

### 15.6 批量模式

针对配角、阵营成员、地点条目等进行批量生成。

## 16. 生成输入设计

### 16.1 输入来源

系统应允许以下输入来源：

- 用户手写概念
- 项目基础设定
- 世界书已有条目
- 已有人物卡
- 已有剧本草稿
- 已有预设

### 16.2 输入控制项

每次生成建议支持：

- 目标资产类型
- 使用模板
- 参考资产
- 风格预设
- 输出详细度
- 是否允许创新扩展

### 16.3 创新边界

辅助生成时应允许一定程度的创造，但不能无限发散。建议采用以下原则：

- 世界书生成优先保守
- 人物卡生成允许适度鲜明化
- 剧本生成允许适度创造冲突
- 预设生成允许适度风格强化

## 17. 输出规范

### 17.1 统一要求

- 输出必须结构完整
- 输出必须字段清晰
- 输出必须可继续编辑
- 输出必须可导出

### 17.2 文本要求

- 避免空泛套话
- 避免重复表达
- 避免设定矛盾
- 尽量具体
- 尽量有实际创作价值

### 17.3 结构字段要求

- `text` 字段必须是单段短文本
- `textarea` 字段必须是可直接阅读的段落
- `json` 字段必须结构合法
- `object_list` 字段必须结构统一

## 18. 校验与质量控制

### 18.1 设计目的

Storyworks 不是只负责“生成”，也必须负责“把生成结果变得更能用”。

### 18.2 基础校验

- 必填字段是否完整
- 字段类型是否正确
- 文本是否明显空泛
- 是否出现明显自相矛盾
- 是否偏离模板要求

### 18.3 世界书校验

- 分类是否正确
- 条目是否表达清楚
- 设定是否具备可用性
- 是否缺少关键背景信息

### 18.4 人物卡校验

- developer_mode 是否完整
- player_mode 是否越界
- 角色背景是否支撑角色形象
- 能力设定是否合理
- 关系信息是否清楚

### 18.5 剧本校验

- 主线是否明确
- 冲突是否存在
- 场景是否有作用
- 信息揭示是否混乱
- 是否缺少关键节点

### 18.6 预设校验

- 风格描述是否清楚
- 偏好是否可执行
- 质量要求是否明确
- 是否存在互相冲突的指令

### 18.7 质量改进方式

建议系统支持：

- 自动指出缺失字段
- 自动建议补充方向
- 自动提示设定冲突风险
- 自动输出优化建议

## 19. 模板系统设计

### 19.1 模板定位

模板是 Storyworks 的核心基础设施之一。没有模板，系统就很容易退化成普通聊天生成器。

### 19.2 模板分类

- 世界书模板
- 人物卡模板
- 剧本模板
- 预设模板

### 19.3 人物卡模板优先级

默认通用角色模板应作为系统内置官方模板之一，后续其他角色模板都应参考它的结构严谨度。

### 19.4 模板能力

模板应至少定义：

- 标签页
- 字段
- 字段类型
- 字段说明
- 填写规则
- 输出格式要求

### 19.5 模板扩展原则

- 允许新增字段
- 允许新增标签页
- 允许针对题材派生模板
- 不应破坏基础结构一致性

## 20. 编辑体验设计

### 20.1 编辑器要求

编辑器应强调“结构清晰、修改方便、可直观查看”。

### 20.2 世界书编辑体验

- 左侧分类
- 中间条目列表
- 右侧条目编辑区

### 20.3 人物卡编辑体验

- 模板标签页切换
- developer_mode / player_mode 切换
- 字段级说明提示
- 一键补全缺失字段

### 20.4 剧本编辑体验

- 大纲视图
- 篇章视图
- 场景卡视图
- 约束信息集中显示

### 20.5 预设编辑体验

- 风格说明区
- 偏好参数区
- 示例输出预览区

## 21. 导出设计

### 21.1 导出目标

Storyworks 生成的资产应方便迁移到其他创作流程中使用。

### 21.2 导出格式

建议支持：

- Markdown
- JSON
- 项目打包导出

### 21.3 导出粒度

- 导出单个世界书条目
- 导出单张人物卡
- 导出单个剧本文档
- 导出单个预设
- 导出完整项目

### 21.4 导出用途

- 给作者继续写作
- 给其他系统继续处理
- 作为资料存档
- 作为模板复用基础

## 22. MVP 范围

第一阶段最小可用版本建议只完成以下闭环：

- 创建项目
- 创建世界书条目
- 使用默认人物卡模板生成角色
- 创建基础剧本
- 创建预设
- 将四类资产导出为 Markdown 或 JSON

## 23. 分阶段实施建议

### 23.1 第一阶段

完成项目基础框架与四类资产最小编辑能力。

### 23.2 第二阶段

完善 AI 生成能力、模板能力和规范化能力。

### 23.3 第三阶段

完善质量校验、导出包和批量生成能力。

### 23.4 第四阶段

完善模板生态、题材模板和更成熟的创作工作流。

## 24. 验收标准

系统达到第一阶段设计目标时，至少应满足：

- 世界书可以创建、编辑和生成
- 人物卡可以基于模板完整生成
- developer_mode 与 player_mode 可正确区分
- 剧本可以形成结构化约束文档
- 预设可以形成明确风格指导
- 四类资产可以导出并继续使用

## 25. 结论

Storyworks 的本质不是“叙事系统平台”，而是“叙事资产辅助生成器”。

它的重点不在主项目架构、服务治理或复杂运行时，而在于帮助创作者把世界观、人物、剧情和风格变成一套真正可用、可改、可存、可导出的资产资料。只要把这四类资产做深、做稳、做规范，Storyworks 就已经具备非常清晰且独立的产品价值。
